# Mathlib API audit for arXiv:2111.08131 (finite-dimensional pilot)

_Archived 2111 note: this audit reflects the finite-dimensional pilot framing in use on 2026-03-07 and is preserved for historical context, not as current Paper2009LDT guidance._

Working assumption for the first pass: stay in finite dimension, model operators as `Matrix d d ℂ` for a finite type `d`, use explicit finite averages, and add a thin local layer for measurements/codes/expander kernels. That matches the project roadmap and avoids premature generality.

Mathlib already has good support for finite-dimensional matrix analysis over `ℂ`; it does **not** appear to have domain-specific APIs for POVMs/submeasurements, linear codes, or weighted expander kernels.

## 1. Matrices over `ℂ`: trace, Frobenius/HS norm, Hermitian, PSD, projections

**Directly reusable**

- Trace: `Matrix.trace`, `Matrix.trace_add`, `Matrix.trace_sum`, `Matrix.trace_conjTranspose`, `Matrix.trace_mul_comm`, `Matrix.trace_mul_cycle`, `Matrix.trace_units_conj` in `Mathlib/LinearAlgebra/Matrix/Trace.lean`.
- Hermitian matrices: `Matrix.IsHermitian`, `Matrix.isHermitian_iff_isSelfAdjoint`, `Matrix.IsHermitian.conjTranspose`, `Matrix.isHermitian_conjTranspose_iff` in `Mathlib/LinearAlgebra/Matrix/Hermitian.lean`.
- Positive semidefinite matrices: `Matrix.PosSemidef`, `Matrix.PosSemidef.isHermitian`, `Matrix.PosSemidef.diag_nonneg`, `Matrix.posSemidef_sum`, `Matrix.posSemidef_conjTranspose_mul_self`, `Matrix.posSemidef_self_mul_conjTranspose`, `Matrix.PosSemidef.trace_nonneg`, `Matrix.trace_conjTranspose_mul_self_eq_zero_iff` in `Mathlib/LinearAlgebra/Matrix/PosDef.lean`.
- Matrix order over `ℂ`: after `open scoped MatrixOrder`, one gets `0 ≤ A` and `A ≤ B` with `Matrix.nonneg_iff_posSemidef` and `Matrix.le_iff` from `Mathlib/Analysis/Matrix/Order.lean`.
- Spectral theorem / eigenvalues for Hermitian matrices: `Matrix.IsHermitian.eigenvalues`, `Matrix.IsHermitian.eigenvectorUnitary`, `Matrix.IsHermitian.spectral_theorem`, `Matrix.IsHermitian.spectrum_real_eq_range_eigenvalues`, `Matrix.IsHermitian.trace_eq_sum_eigenvalues` in `Mathlib/Analysis/Matrix/Spectrum.lean`.
- Positivity via eigenvalues: `Matrix.IsHermitian.posSemidef_iff_eigenvalues_nonneg`, `Matrix.PosSemidef.eigenvalues_nonneg` in `Mathlib/Analysis/Matrix/PosDef.lean`.
- Functional calculus / absolute value / square root: generic `CFC.sqrt`, `CFC.abs`, `CFC.abs_sq` and matrix-facing lemmas such as `Matrix.PosSemidef.sq_sqrt`, `Matrix.PosSemidef.posSemidef_sqrt`, `Matrix.IsHermitian.cfc`, `Matrix.IsHermitian.cfc_eq` in `Mathlib/Analysis/Matrix/Order.lean`, `Mathlib/Analysis/Matrix/HermitianFunctionalCalculus.lean`, and `Mathlib/Analysis/SpecialFunctions/ContinuousFunctionalCalculus/Abs.lean`.
- Frobenius norm: with `open scoped Matrix.Norms.Frobenius`, use `Matrix.frobenius_norm_def`, `Matrix.frobenius_norm_conjTranspose`, `Matrix.frobenius_norm_mul` from `Mathlib/Analysis/Matrix/Normed.lean`.
- If operator norm / adjoints are needed later: `Matrix.toEuclideanLin`, `Matrix.toEuclideanLin_conjTranspose_eq_adjoint` in `Mathlib/Analysis/InnerProductSpace/PiL2.lean` and `Mathlib/Analysis/InnerProductSpace/Adjoint.lean`; `open scoped Matrix.Norms.L2Operator` gives the $\ell^2$ operator norm via `Mathlib/Analysis/CStarAlgebra/Matrix.lean`.

**Thin local wrappers worth adding**

- `abbrev Op (d) := Matrix d d ℂ`.
- Normalized trace `tr : Op d → ℂ`, since Mathlib has `Matrix.trace` but no paper-style normalized trace wrapper.
- Hilbert--Schmidt / $\tau$-norm wrapper, preferably built from Frobenius norm: e.g. `hsNorm A := (Fintype.card d : ℝ)^(-1/2) * ‖A‖` under `Matrix.Norms.Frobenius`.
- A local lemma identifying the Frobenius norm with trace data, e.g. `hsNorm_sq_eq_re_trace_conjTranspose_mul`; I did **not** find this already packaged.
- A local predicate for projectors, e.g. `IsProj A := A.IsHermitian ∧ A * A = A` (or `A ^ 2 = A`). I did not find a dedicated `Matrix` projection predicate.

**Recommendation for the first pass**

- Use `Matrix d d ℂ` throughout Sections 2--3.
- Use Frobenius norm as the finite-dimensional stand-in for the paper’s $\tau$-norm.
- Use the matrix order (`MatrixOrder`) for positivity statements.
- Delay any trace-norm / Schatten-$1$ API: I did not find dedicated Mathlib support for `traceNorm`, “Schatten”, or “nuclear” norms.

## 2. Finite families of operators; measurements and submeasurements

**Directly reusable**

- Finite sums/products over answer and question sets are standard `Fintype`/`Finset` big operators.
- `Matrix.posSemidef_sum` is exactly the lemma needed to sum positive effects.
- `Matrix.trace_sum` is useful for probabilities and normalization checks.
- Fiberwise regrouping for processed measurements is already there: `Finset.sum_fiberwise`, `Finset.sum_fiberwise_of_maps_to`, `Fintype.sum_fiberwise` in `Mathlib/Algebra/BigOperators/Group/Finset/Basic.lean`.
- For product answer sets or nested indexing, `Finset.sum_product` and `Finset.sum_sigma` from `Mathlib/Algebra/BigOperators/Group/Finset/Sigma.lean` are handy.
- If one later switches from matrices to endomorphisms/subspaces, projection APIs already exist: `LinearMap.IsProj` in `Mathlib/LinearAlgebra/Projection.lean`, and `LinearMap.IsSymmetricProjection`, `Submodule.starProjection`, `Submodule.orthogonalProjection` in `Mathlib/Analysis/InnerProductSpace/Symmetric.lean` and `.../Projection/Basic.lean`.

**What is missing**

- I did not find any dedicated POVM / projective-measurement / submeasurement API in Mathlib.

**Thin local wrappers worth adding**

- `structure Submeasurement (α d)` with fields `M : α → Op d`, `pos : ∀ a, 0 ≤ M a`, `sum_le_one : ∑ a, M a ≤ 1`.
- `structure Measurement ...` replacing `≤ 1` by `= 1`.
- `ProjectiveMeasurement` or a field `isProj : ∀ a, IsProj (M a)`.
- `process (f : α → β)` defined by fiberwise sums; this is exactly where `sum_fiberwise` will pay off.
- Repo-local definitions of consistency/closeness on question-indexed families `X → α → Op d`.

**Recommendation for the first pass**

- Keep this layer repo-local; do not try to force an abstract probability or category-theoretic interface yet.
- Index measurements as raw functions `α → Op d` and question families as `X → α → Op d`.
- Build data processing lemmas on top of `sum_fiberwise` immediately; those are used everywhere in the preliminaries.

## 3. Function spaces on finite types, submodules of functions, and linear codes

**Directly reusable**

- Standard basis of function spaces: `Pi.basisFun` in `Mathlib/LinearAlgebra/StdBasis.lean`.
- Coordinate maps: `LinearMap.proj`, `LinearMap.single`, `LinearMap.pi`, `LinearMap.ker_pi` in `Mathlib/LinearAlgebra/Pi.lean`.
- Submodules of function spaces: `Submodule.pi`, `Submodule.mem_pi`, `Submodule.map`, `Submodule.comap`.
- Dimensions/cardinalities: `Module.finrank_fintype_fun_eq_card`, `Module.finrank_matrix` in `Mathlib/LinearAlgebra/Dimension/Constructions.lean`; `Fintype.card_fun` in `Mathlib/Data/Fintype/BigOperators.lean`.
- If generator matrices are later used: `Matrix.rank_eq_finrank_span_cols`, `Matrix.rank_eq_finrank_span_row` in `Mathlib/LinearAlgebra/Matrix/Rank.lean`.

**What can be modeled cleanly**

- A linear code over a field `𝔽` as a `Submodule 𝔽 (ι → 𝔽)`.
- Coordinate restriction maps as linear maps built from `LinearMap.pi` and `LinearMap.proj`.
- Tensor-code constraints as intersections of `Submodule.comap`s of line-restriction maps.

**Thin local wrappers worth adding**

- `abbrev Word (ι 𝔽) := ι → 𝔽`.
- `structure Code` bundling a `Submodule 𝔽 (ι → 𝔽)` together with distance metadata.
- `restrictCoords (s : Finset ι) : Word ι 𝔽 →ₗ[𝔽] (↥s → 𝔽)`.
- `lineRestrict` maps for `Fin m → Fin n` words; use `Function.update` to model axis-parallel lines.
- `Interpolable` as existence of a linear inverse/section to coordinate restriction on every `t`-subset.

**Recommendation for the first pass**

- Use `Fin n` for coordinates and `Fin m → Fin n` for `[n]^m`.
- Define tensor codes combinatorially via line restrictions, not via algebraic tensor products. Mathlib has tensor-product machinery, but the paper’s notion is closer to an intersection of local constraints.
- Keep generator matrices optional; they are useful later for equivalence statements, but not required for the initial code API.

## 4. Hamming distance

**Directly reusable**

- `hammingDist`, `hammingNorm`, `hammingDist_eq_zero`, `hammingDist_pos`, `hammingDist_le_card_fintype`, `hammingDist_eq_hammingNorm`, `hammingDist_comp`, `hammingDist_smul` in `Mathlib/InformationTheory/Hamming.lean`.
- The `Hamming` type synonym is also available there if one wants an actual metric-space / normed-group structure.

**Thin local wrappers worth adding**

- Probably only a normalized relative-distance wrapper, e.g. `relDist : Word ι 𝔽 → Word ι 𝔽 → ℚ` or `ℝ`.
- For the code API, a proposition like `HasMinDistance (C : Code ...) (d : ℕ)` is likely enough.

**Recommendation for the first pass**

- Use `hammingDist` directly; there is no need to re-define it locally.
- The plain raw-function version is likely simpler than the `Hamming` type synonym for early coding-theory lemmas.

## 5. Finite graphs, Laplacians, eigenvalues, Rayleigh quotient, appendix expander lemma

**Directly reusable**

- Simple graphs: `SimpleGraph.adjMatrix` in `Mathlib/Combinatorics/SimpleGraph/AdjMatrix.lean`.
- Laplacian matrix API: `SimpleGraph.lapMatrix`, `SimpleGraph.lapMatrix_mulVec_apply`, `SimpleGraph.lapMatrix_mulVec_const_eq_zero`, `SimpleGraph.lapMatrix_toLinearMap₂'`, `SimpleGraph.posSemidef_lapMatrix`, `SimpleGraph.card_connectedComponent_eq_finrank_ker_toLin'_lapMatrix` in `Mathlib/Combinatorics/SimpleGraph/LapMatrix.lean`.
- Spectral tools for Hermitian matrices are already strong enough to diagonalize local Laplacians: `Matrix.IsHermitian.spectral_theorem`, `...eigenvalues`, `...spectrum_real_eq_range_eigenvalues`.
- General Rayleigh quotient API for self-adjoint operators: `ContinuousLinearMap.rayleighQuotient` and `LinearMap.IsSymmetric.hasEigenvalue_iSup_of_finiteDimensional` / `...iInf...` in `Mathlib/Analysis/InnerProductSpace/Rayleigh.lean`.
- Weighted transition kernels: `Matrix.rowStochastic`, `Matrix.colStochastic`, `Matrix.mem_rowStochastic_iff_sum`, `Matrix.mem_colStochastic_iff_sum`, `Matrix.sum_mulVec_of_mem_colStochastic` in `Mathlib/LinearAlgebra/Matrix/Stochastic.lean`.

**Important limitations**

- `SimpleGraph` is for unweighted simple graphs, but the appendix is stated for an **edge distribution** with uniform marginals; that is more naturally a weighted kernel `K : Matrix V V ℝ` (or `ℂ`).
- `SimpleGraph.lapMatrix` is the unnormalized combinatorial Laplacian, while the paper uses the normalized operator `L = (1/n) I - K`.
- `SimpleGraph.posSemidef_lapMatrix` is proved over trivially starred ordered fields, not directly in the `Matrix n n ℂ` order language used elsewhere.
- I did not find a ready-made theorem of the exact form `L ≥ λ₂ • (1 - |φ₀⟩⟨φ₀|)` for weighted kernels.

**Thin local wrappers worth adding**

- A local `structure EdgeKernel V` or `ExpanderKernel V` bundling:
  - `K : Matrix V V ℝ` (or `ℂ`),
  - symmetry/Hermitianity,
  - entrywise nonnegativity,
  - uniform row and column sums.
- `laplacian : Matrix V V ℝ := (Fintype.card V : ℝ)⁻¹ • 1 - K`.
- A local spectral-gap field or hypothesis expressing the lower bound on nontrivial eigenspaces.
- A rank-one projector onto the all-ones vector, if needed in the exact paper form.

**Recommendation for the first pass**

- For the appendix lemma, do **not** start from `SimpleGraph` unless you later specialize to regular unweighted graphs.
- Use a local weighted-kernel API and prove the required inequality with matrix spectral theory.
- If the proof is written operator-theoretically, use `Matrix.toEuclideanLin` / `toEuclideanCLM` only as a bridge; the matrix proof should still be the most direct.

## 6. Averages / expectations over finite types and finite sets

**Directly reusable**

- Raw finite averaging ingredients: `Finset.univ`, `Finset.card_univ`, `Finset.sum_product`, `Finset.sum_sigma`, `Fintype.card_prod`, `Fintype.card_fun`.
- Uniform finite distributions: `PMF.uniformOfFintype`, `PMF.uniformOfFintype_apply` in `Mathlib/Probability/Distributions/Uniform.lean`.
- Nonuniform finite distributions: `PMF.map` in `Mathlib/Probability/ProbabilityMassFunction/Constructions.lean`, `PMF.bind` in `.../Monad.lean`, `PMF.toMeasure` in `.../Basic.lean`, and `PMF.integral_eq_sum` in `.../Integrals.lean`.
- Full measure-theoretic average exists as `MeasureTheory.average` in `Mathlib/MeasureTheory/Integral/Average.lean`.

**Thin local wrappers worth adding**

- `avg (f : α → β) := (Fintype.card α : ℝ)⁻¹ • ∑ a, f a` for additive codomains.
- `avg₂` and similar wrappers for product distributions.
- `expect (p : PMF α) (f : α → β)` when nonuniform distributions actually appear.

**Recommendation for the first pass**

- For Sections 2--3, explicit `Fintype` averages are simpler than `MeasureTheory.average`.
- Use `PMF` only when the distribution is genuinely nonuniform (most notably the appendix edge distribution).
- Keep all averaging notation repo-local; this will hide coercions and repeated `Fintype.card` algebra.

## Concrete first-pass recommendation

1. Fix a reusable operator layer around `Matrix d d ℂ` with normalized trace, Frobenius/HS norm, positivity, and projector predicates.
2. Add a local measurement/submeasurement API on finite answer types.
3. Model linear codes as submodules of function spaces, with coordinate restriction maps as explicit linear maps.
4. Use `hammingDist` directly for distance.
5. Treat the appendix expander object as a weighted symmetric kernel, not as a `SimpleGraph`, unless a later specialization justifies the detour.
6. Keep averages explicit and finite; introduce `PMF` only where nonuniformity matters.
